# 游戏 SERP 可填充草稿系统

## 概览

这个 skill 用来把游戏关键词从“想法”变成“70% 到 85% 已完成的页面草稿”。草稿必须接近可发布，但保留清晰的人工补位槽位，方便你补截图、验证、版本确认、路线图、证据和最终判断。

这个 skill 只负责单页生产，不负责整站治理。首页层级、Hub 结构、复查计划、站外验证和上线后动作树，交给 `$new-game-site-sop` 处理。

这个 skill 的另一条硬规则是：内部工作流和最终页面必须分层。草稿里允许有占位符、证据备注和编辑任务；最终页面里不允许出现这些内部痕迹。

## 用户页写作目标

最终页面必须像成熟玩家写给玩家的攻略：

- 直接回答问题
- 直接给步骤、比较、路线和建议
- 只保留用户需要知道的背景
- 不暴露 AI、编辑、调研、验证、判断过程
- 不写给搜索引擎看，只写给玩家看

## 内部流程排除清单

下列内容只能留在草稿、备注、占位符、CMS 字段或发布检查单中，不能进入最终页面正文：

- “我查了什么资料后得出这个结论”
- “这里还需要验证 / 判断 / 确认”
- “以下是调研结果 / SERP 发现 / 证据分层”
- “这个关键词值得做 / 不值得做”
- “为了 SEO / 排名 / 覆盖关键词”
- “人工补充后再发布”
- “这是 AI 起草的内容”
- 任何占位符原文

如果这些内容对玩家真的有帮助，必须改写成玩家视角。例如：

- 不写“这里仍需验证掉率”
- 改写为“当前版本社区常见反馈是 X，稳定样本还不多，因此更适合把它当成低概率掉落看待”

## 用户可见内容保留清单

最终页面应优先保留：

- 先说结论的快速答案
- 玩家立刻能执行的步骤
- 玩家做选择时真正关心的比较
- 条件、限制、适用场景
- 失败点、绕路成本、常见误区
- 地图、截图、表格、路线图
- 与当前问题强相关的 FAQ

## 案例优先与一手经验优先

如果同一个主题既可以写成抽象教程，也可以写成真实案例、真实场景或真实路径，默认优先后者。

优先级顺序：

1. 真实案例
2. 一手经验
3. 真实场景化例子
4. 官方事实整理
5. 二手资料拼装

规则：

- 能用真实开局、真实路线、真实取舍说明，就不要先写泛步骤
- 能用自己或站点的一手观察说明，就不要把公开资料改写当主体价值
- 二手资料只适合作为补充、校正和覆盖，不适合作为页面的核心卖点

## 工作流

### 1. 先做多维 SERP 调研，不允许只搜一轮

至少覆盖这些调研层：

- 官方层：官网、patch notes、开发者帖子、平台商店页
- 排名层：当前 SERP 前排攻略站和 Wiki
- 社区层：Reddit、Steam 讨论、Discord 摘要、论坛
- 创作者层：YouTube 视频、视频描述、可用转录
- 词形扩展层：同义词、缩写、错拼、玩家黑话
- 邻近意图层：`best`、`how to`、`where to find`、`worth it`、`build`、`unlock`、`fix`

规划和审稿时读取 `references/serp-research-matrix.md`。

### 2. 强制做 Query Intent Classification

先把查询分类，不要直接开写。

使用下面的分类：

#### ACTION
- how to unlock
- how to get
- where to find
- how to fix

#### DECISION
- best build
- worth it
- tier list
- comparison

#### REFERENCE
- wiki
- database
- codex
- drop rate

#### NAVIGATION
- map
- route
- spawn location

#### UPDATE
- patch notes
- buffs
- nerfs

不同 intent 必须触发不同页面结构。未完成分类前不要起草正文。

### 3. 先做实体提取，避免 AI 乱编

写正文前先抽取实体：

- 游戏名
- 职业 / 流派
- 武器 / 装备
- Boss / NPC
- 地图 / 区域
- 材料 / 资源
- 机制 / 状态 / 词条
- 补丁版本

输出格式示例：

```yaml
实体:
  游戏: Mistfall Hunter
  职业: Fire Mage
  Build: PvE Endgame
  技能:
    - Flame Burst
    - Fire Nova
  武器:
    - Dragon Staff
```

### 4. 做 SERP Opportunity Score，先判断值不值得做

对每个候选页打分：

```yaml
SERP 缺口评分:
  覆盖缺口: 0-5
  证据缺口: 0-5
  体验缺口: 0-5
  工具缺口: 0-5
  时效缺口: 0-5
  视觉缺口: 0-5
  总机会分: 0-30
```

如果总机会分很低，不要浪费时间做同质化页面。

如果你只是需要“该不该做这页”的判断，不要继续写满草稿，先停在评分和页面角度阶段。

### 5. 做 Evidence Confidence Layer

对关键结论强制打证据等级：

```yaml
证据等级:
  A: 官方来源
  B: 多个官方 + 社区交叉支持
  C: 多个社区共识
  D: 单一创作者或单一来源
  F: 猜测或未证实
```

不要只写“可能”“大概”。要明确写出证据等级。
证据等级只服务内部写作和发布判断。最终页面不要直接向玩家展示 A/B/C/F 这种工作流标签，除非页面本身就是在做数据可信度对比。

如果页面的关键结论缺少一手经验或真实案例支撑，就不要伪装成“我已经验证过的经验结论”。这时要把页面降级为整理页、资料页或观察页。

### 6. 做 Patch Drift System

游戏页天然有版本漂移风险，必须标记稳定度：

```yaml
补丁稳定性:
  STATIC: 世界观、固定地图、静态 NPC
  SEMI-STABLE: 材料、配方、一般掉落
  VOLATILE: Build、Meta、Tier List、平衡性
```

并强制给出复查周期：

- `VOLATILE`：14 天
- `SEMI-STABLE`：30 到 60 天
- `STATIC`：180 天

### 7. 判断页面应该是文章还是工具

不要默认所有关键词都产出文章。强制判断：

```yaml
页面候选类型:
  - 文章
  - 数据库
  - 计算器
  - Build Builder
  - Tracker
  - Planner
  - Generator
  - Simulator
```

如果查询明显更适合工具，就不要勉强写成长文攻略。

### 8. 规划资产层

不仅要写 `截图`，还要明确资产种类：

```yaml
所需资产:
  - 头图
  - 地图截图
  - 定位标记图
  - 掉落表
  - Build 表
  - 对比表
  - 视频时间戳
  - Patch 引用
  - 交互小工具
```

不要把所有资产都强行塞进一页。只保留对当前 intent 真有帮助的资产。

### 9. 写出接近完成的页面，但保留人工补位槽位

草稿必须接近完整页面，而不是只给提纲。至少要包含：

- 标题方向
- H1
- 引言
- 快速答案
- 核心正文
- 表格
- FAQ
- 内链规划

但人工补位部分要保留占位符，不要强行写死。
最终交付给用户前，所有阻断型占位符都必须消失或被改写成自然的人话内容。

## 中文占位符格式

页面中统一使用这些占位符：

- `[[截图: 需要什么图片]]`
- `[[验证: 需要人工确认什么说法]]`
- `[[来源: 还需要补什么来源]]`
- `[[判断: 这里需要人工做推荐选择]]`
- `[[可选证据: 有更好，但不是上线阻断项]]`
- `[[实测: 需要人工实测什么]]`
- `[[补丁: 需要确认哪个 patch 版本]]`
- `[[社区共识: 需要补哪个社区讨论]]`
- `[[视频证据: 需要补哪个视频时间戳]]`
- `[[数据挖掘: 需要补哪个 datamine 结论]]`

规则：

- 占位符必须短、准、可执行
- 一次补充动作就能完成一个占位符
- 一篇普通页面不应留下超过 3 到 8 个关键占位符
- 如果一节里面全是占位符，说明调研还不够

占位符只允许存在于内部草稿版本，不能进入最终页面。

## 页面写作规则

- 先给答案，再给背景
- 只有在能降低决策成本时才用表格
- 不确定的内容必须标出来，不能抹平
- 【事实】和【推荐】必须分开
- 不要用空泛开头和“最全 wiki”承诺灌水
- 证据弱的地方，不要假装确定
- 如果一个 section 去掉游戏名后仍然像通用模板段落，重写它

进一步强制：

- 不要写“本文将”“下面我们来”“综合来看”这类空转句
- 不要写 SEO 术语、内容生产术语、编辑术语
- 不要把调研过程、验证过程、判断过程写给玩家
- 不要连续制造机械化 H2/H3，仅保留能帮助扫描的标题
- 不要用 FAQ 填补正文信息不足
- 不要为了关键词覆盖重复说同一个结论
- 每个段落都要服务一个玩家动作：理解、选择、执行、避坑
- 对 `DECISION`、`ACTION`、`NAVIGATION` 页，默认给出站点自己的推荐或路线判断，不要只做中立转述
- 默认写出至少一个失败条件、例外或不适用场景
- 如果页面主要价值来自真实案例，就先写案例，再抽象出结论
- 如果没有案例和一手经验，就不要硬写成“我是这么做的”口吻

## 作者判断规则

区分两类页面：

- `资料页`：以事实、来源、定义、掉落、配方为主，可以降低判断强度
- `决策页`：以最佳、推荐、是否值得、路线、Build、优先级为主，必须给出明确判断

对于决策页，默认输出：

- 推荐什么
- 不推荐什么
- 适合谁
- 不适合谁
- 为什么这样判断

禁止用这些句子替代判断：

- “这取决于你的玩法”
- “建议根据个人需求选择”
- “不同玩家有不同答案”
- “没有绝对最优”

除非页面的结论本身就是“当前版本不存在稳定共识”，否则不要用这些句子逃避判断。

## 失败与边界规则

每篇普通页面至少补 1 类边界信息：

- 什么条件下这套方法失效
- 什么人不适合照做
- 哪个版本前提改变后要重看
- 哪个常见社区说法现在已经不适用
- 哪个看起来更快的方法实际更亏

如果一个页面没有任何边界、例外、失败条件，它大概率仍然太像模板稿。

## 用户需求对齐规则

新增和更新都必须先回答这 4 个问题，再动笔：

1. 玩家搜索这个词时，最想立刻知道什么？
2. 玩家看完这一页后，应该能做出什么动作或决策？
3. 玩家最容易卡住、选错或浪费时间的地方是什么？
4. 哪些信息如果不写，会导致这页虽然“完整”但对玩家没用？

如果一个新增段落、一个更新说明或一个 FAQ 不能明显提升上面 4 点中的至少 1 点，就不要加。

## 反模板与反 AI 痕迹检查

发布前逐条检查：

1. 首屏是否已经给出答案，而不是铺垫背景？
2. 去掉实体名后，段落是否还能复用到别的游戏页？
3. 是否出现了“看起来正确，但对玩家没帮助”的空话？
4. 是否出现“工作流泄漏”，比如验证、判断、确认、调研、关键词、SEO？
5. 是否存在整段 FAQ、整段背景、整段定义，只是为了拉长页面？

只要第 2、3、4、5 任意一项为“是”，就重写。

单页反模板评分卡读取 `references/anti-template-page-scorecard.md`。

## 更新内容规则

这个 skill 不只用于新页，也用于旧页更新。更新时默认执行下面规则：

- 不要只补版本号、日期和一两句套话
- 先检查玩家结论是否变了，再决定改哪一节
- 如果 patch 改到了获取方式、Build 推荐、优先级、路线成本，就必须改正文结论
- 如果只改后台证据，不改玩家内容，不算一次有效更新
- 更新后的页面仍要重新产出“用户可见纯净版”
- 如果页面原本依赖真实案例或一手经验，更新时要确认这些案例是否仍成立，而不是只改 patch 编号

更新页时，优先回答：

1. 玩家现在最关心的是“还能不能按旧方法做”还是“现在该改什么”
2. 哪个旧结论已经不再适用
3. 哪个步骤、比较或推荐必须重写
4. 有没有新增一个更直接的用户答案，而不是只新增编辑注记
5. 原来的失败边界和不适用场景是否也变了

## 内链规划器

不要只给“建议链接”。要输出结构化内链：

```yaml
内链规划:
  父页:
  子页:
  兄弟页:
```

示例：

```yaml
内链规划:
  父页: Fire Mage
  子页:
    - Fire Mage Build
    - Fire Mage Skills
    - Fire Mage Rotation
    - Fire Mage Gear
  兄弟页:
    - Ice Mage
    - Lightning Mage
```

## 信息密度检查

AI 长文最容易水。每次都输出：

```yaml
信息密度检查:
  事实数:
  决策点:
  表格数:
  可执行步骤数:
  灌水段落数:
```

如果灌水段落占比大于 15%，重写。

## 输出结构

使用这个 skill 时，按顺序输出 10 个区块：

1. `查询分类`
2. `实体提取`
3. `SERP 发现`
4. `SERP 缺口评分`
5. `页面候选类型`
6. `页面角度`
7. `补丁稳定性`
8. `资产规划`
9. `草稿`
10. `人工补位清单`
11. `反模板评分`

### 查询分类

输出：

- intent 类型
- 用户真正要解决的问题
- 页面应该采用什么模板

### 实体提取

输出结构化实体。

### SERP 发现

总结：

- 主要 intent cluster
- 竞品常见模式
- 竞品薄弱点
- 找到的官方事实
- 有争议或低置信度部分

### 页面角度

输出：

- 目标 query
- 页面类型
- 为什么这个角度优于普通攻略
- 上线前必须补什么人工证据

### 草稿

写出接近完整的页面正文，并嵌入中文占位符。

草稿阶段可以保留内部标记，但在生成最终可发布版本时，必须额外输出一份“用户可见纯净版”：

- 去掉占位符
- 去掉工作流术语
- 去掉证据分层标签
- 去掉任何 AI 或编辑过程表述
- 保留玩家真正需要的内容

如果页面属于 `真实案例页` 或 `经验决策页`，草稿里还要明确标记：

- 哪一段来自真实案例
- 哪一段来自一手经验
- 哪一段只是事实整理

最终纯净版不展示这些标签，但要保留这些内容层次。

### 人工补位清单

把所有占位符整理成 checklist，便于你逐项补完。

### 反模板评分

必须输出：

- 单页反模板总分
- 最危险的 3 个模板化信号
- 是否允许直接进入最终纯净版
- 如果不允许，先重写哪 1 到 2 个部分
- 当前页面属于哪种内容来源等级：`S / A / B / C`

## 发布门槛

判断页面可否上线前，确认：

- 页面只解决一个明确 query 或意图簇
- 至少规划了一个非模板资产
- 推荐结论部分有证据或 `[[判断: ...]]`
- 高风险说法有 `[[验证: ...]]`
- 来源敏感部分有 `[[来源: ...]]`
- 有补丁漂移风险的内容写明复查周期
- 最终页面里不存在任何内部流程用语
- 最终页面从开头到结尾都站在玩家视角，而不是编辑或 SEO 视角
- 单页反模板评分达到可发布阈值
- 如果页面是决策页，必须有明确推荐、不推荐和边界
- 如果页面声称来自经验，必须有案例或一手材料支撑

如果这页只要把游戏名替换一下就能套到别的站上，说明角度还不够，先重写再发布。

默认阈值：

- `80 分及以上`：可发布
- `65-79 分`：需重写危险区块后再发布
- `64 分及以下`：默认模板化高风险，不发布


如果需要更细的 v2 模块定义，读取 `references/game-serp-v2-modules.md`。
